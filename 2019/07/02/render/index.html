浏览器渲染机制

发表于 2019-07-02 | 分类于 -前端 |
字数统计:

概述

我们写好代码，打包完成，上线以后，浏览器是如何将页面呈现呢？

本文分析浏览器解析文件及其渲染过程。

基本概念

在学习具体渲染过程之前，需要先了解几个具体概念。

进程和线程

进程是CPU进行资源分配的基本单位。

进程会占用系统资源，在一个进程内可以存在一个或多个线程，这就是单线程和多线程，但是无论是单线程还是多线程，都是在一个进程内。

JS是一个单线程的语言。

线程是CPU调度的最小单位，是建立在进程的基础上的运行单位，共享进程的内存空间。

浏览器是多进程的，不同类型的标签页都会开启一个新的进程，相同类型的标签页会合并成一个进程。

chrome中的任务管理器可以看见标签页的进程ID。

浏览器进程

浏览器进程分为四个：

浏览器进程

  • 负责管理各个标签页的创建与销毁
  • 负责浏览器的界面显示和功能（前进，后退，收藏等）
  • 负责资源的管理与下载

GPU进程

  • 负责3D绘制和硬件加速

第三方插件进程

  • 负责每个第三方插件的使用
  • 每个第三方插件使用时候都会创建一个对应的进程

*浏览器渲染进程

  • 浏览器内核
  • 主要负责HTML，CSS，JS等文件的解析和执行

浏览器结构

用户界面

浏览器主页面，包括地址栏、后退前进按钮、书签栏、主页按钮、刷新停止按钮等，除下半部分请求页面之外的所有内容。

浏览器引擎

协调UI和渲染引擎的工作

渲染引擎

展示请求内容，html\css等

网络组件

负责网络调用，例如 HTTP 请求等，使用一个平台无关接口，下层是针对不同平台的具体实现

JS引擎

解析执行JS代码

UI后端

绘制基本UI组件，暴露一个统一的和平台无关的接口，下层使用操作系统的UI方法实现。

数据存储

持久层

浏览器可能需要在本地存储各种各样的数据，例如 Cookie 等。浏览器也需要支持诸如 localStorage，IndexedDB，WebSQL 和 FileSystem 之类的存储机制

浏览器渲染过程

浏览器渲染进程在渲染过程中，会开启多个线程一同工作：

  1. GUI渲染线程
    • 负责解析HTML文件构建DOM树，解析CSS结合DOM树渲染成RenderObject树，然后布局和绘制页面
    • 当RenderObject树需要更新样式属性时，即发生重绘（Repaint）或RenderObject树中的元素规模尺寸，布局或显示隐藏等发生改变，即发生回流（reflow）时执行
  2. JS引擎线程
  3. 事件触发线程,
  4. 定时器触发器线程
  5. 异步http请求线程

注：GUI渲染线程与JS引擎线程是互斥的，因为JS引擎线程在执行过程中可能会发生重绘和回流，所以GUI渲染线程执行时候，JS引擎线程会被挂起，等待GUI渲染线程执行完毕才会执行；JS引擎线程执行时候同理

接收数据以后：

HTML解析

  • 构建DOM树

CSS解析

  • 每个CSS文件都被解析成一个样式表对象，CSSOM

页面渲染

  • 遍历节点，创建渲染树（Render tree)
  • 自底向上计算每个节点首选宽度
  • 自顶向下给每个节点分配实际宽度
  • 自底向上的计算每个节点的高度
  • 根据计算结构构建节点坐标，布局渲染树Layout
  • 绘制渲染树，绘制页面的像素信息Paint
  • 浏览器主进程将默认图层和复合图层交给GPU进程

以上都是GUI渲染线程过程。

GPU渲染

  • 将各个图层合成（composite），最后显示出页面

渲染结束

  • 根据JS文件里的DOM操作等行为，可能重新产生又一轮的渲染和绘制

Reflow

浏览器为了重新渲染部分或整个页面，重新计算页面元素位置和几何结构的进程叫做reflow.

页面上任何一个节点触发了reflow，会导致它的子节点及祖先节点重新渲染。所以我们了解如何减少reflow次数，及DOM的层级，css效率对refolw次数的影响是十分有必要的。

什么时候会导致reflow发生呢？

  • 改变窗口大小
  • 改变文字大小
  • 添加/删除样式表
  • 内容的改变，(用户在输入框中写入内容也会)
  • 激活伪类，如:hover
  • 操作class属性
  • 脚本操作DOM
  • 计算offsetWidth和offsetHeight
  • 设置style属性

Repaint

repaint是指

一个元素的外观被改变，但没有改变布局的情况下发生的，如改变了visibility、outline、background等。当repaint发生时，浏览器会验证DOM树上所有其他节点的visibility属性。

通俗来说，就是当各种盒子的位置、大小以及其他属性，例如颜色、字体都确定下来后，浏览器便把这些元素都按照各自的特性绘制一遍，于是页面的内容出现了，这个过程叫做repaint

渲染优化

  • 尽可能提前引入css文件，例如在头部引入css文件;
  • 尽可能早加载css文件中引用的资源，例如自定义字体文件，可以使用预加载，在link标签加入’rel=”preload” as=”font”‘该元素属性,不然会造成渲染阻塞
  • 在DOM和CSS渲染之后加载js文件，例如在尾部加载js文件，或者使用该元素属性”defer”和”async”,进行js文件异步加载，但是在不同浏览器会有兼容性问题。

CSS阻塞渲染

  • 媒体查询，根据media给到不同渲染

  • preload预加载

  • 动态添加link

  • 将CSS放在head，不管内联还是外联都尽早开始下载或者构建CSSOM（前提是这个CSS是首屏必须的

    • IE 只要看到HTML 标签就会进行绘制
    • chrome 不管css放在前面还是后面，都要等到CSSOM构建形成后才会绘制到页面上
    • firefox 放在head则会阻塞绘制，放在body末尾会先绘制前面的标签

  • 避免使用CSS import

  • 适度内联CSS

  • 不使用CSS计算，避免使用通配、高级选择器

    • 高消耗属性：
      • box-shadows
      • border-radius
      • transparency
      • transforms
      • CSS filters（性能杀手）

JS阻塞渲染

由于DOM解析的时候，遇见JS文件，会先解析加载JS，直至解析完成，才会继续往下解析DOM，这被称为JS阻塞渲染。

  • 对于动画效果的实现，避免使用 setTimeout 或 setInterval，请使用 requestAnimationFrame。
  • 将长时间运行的 JavaScript 从主线程移到 Web Worker。
  • 使用微任务来执行对多个帧的 DOM 更改。
  • 使用 Chrome DevTools 的 Timeline 和 JavaScript 分析器来评估 JavaScript 的影响。

相关问题

为什么要将JS放置到页脚？

  1. JS放在页头的话，会暂停GUI渲染线程，暂停页面渲染，调用JS引擎线程先解析加载JS，就会导致JS阻塞渲染。
  2. 因为JS中可能会进行一些DOM操作，放在页头的话，可能当前被操作DOM还没加载出来，会导致报错。

为什么要将CSS放置到HEAD中？

  1. CSS文件放在顶部一方面是因为放置顺序决定了下载的优先级，更关键的是浏览器的渲染机制。
  2. 当它放在底部时，需要等所有的HTML和JS全部解析完成之后才开始下载，会造成样式卡顿。
  3. 当CSSOM没构建完成的时候，页面不会进行渲染。
  4. CSS 不会阻塞 DOM 的解析，但会阻塞 DOM 渲染。

整个前端性能提升大致分几类？

  1. 静态资源优化
    1. 合并css、js文件，制作雪碧图：减少http的请求次数，节省网络请求时间
    2. 静态资源cdn分发：客户端可以通过最佳的网络链路加载静态资源
    3. js、css文件压缩，图片压缩，gzip压缩：减少请求返回的数据量
    4. 合理设置HTTP缓存机制
    5. 权衡dns的查找
  2. 接口访问优化
    1. 首屏直出、同构
      1. 如果页面的初始化数据，在后端完成渲染，其它的用户交互使用ajax的方式完成，也就是传统意义上的首屏直出，就可以得到很好的解决
      2. 缺点：前端后台要维护两套模板
      3. 但node出来以后，前后端都都可以使用js语言
      4. react在同构直出方面做得比较出众
    2. 接口合并
    3. 迁移至HTTP/2 可解决 HTTP/1.1 的许多固有性能问题，例如并发请求限制和缺乏标头压缩
    4. 关键资源预加载，rel=preload
    5. 延迟加载非关键资源（图像，视频）
  3. 页面渲染速度优化
    1. css放在顶部：优先渲染
    2. js放在底部：避免阻塞
    3. 减少DOM元素数量：这个最能体现变成水平了
    4. img标签要设置高宽：减少重绘重排
    5. 缩小样式计算的范围和复杂性
    6. 避免布局抖动或者太复杂的布局

关键渲染路径

  1. DOM -> CSSOM -> Render Tree -> Layout -> Paint
  2. js修改时，从DOM或CSSOM开始有一次渲染绘制。

首屏加载优化

最优的首屏加载时间在2s之内

  1. 首屏最小化，控制DOM节点数、请求数、外链数
  2. 首屏数据拉取逻辑置于顶部，数据能够第一时间返回
  3. 首屏渲染css及js逻辑优先内联HTML，当HTML文档返回时CSS和JS能够立即执行。注意的是HTML文档最好不要超过14kb。这是由于，TCP协议里面有一个流控机制，被称为 slow start，也就是在连接建立过程中逐渐增加传输的分段(segments)大小，根据Paul Irish在他的演讲“Delivering the Goods”给出的结论是，一个Web应用最开始的 14kb 数据是最重要的。
  4. 次屏逻辑延后处理和执行，各种数据上报最好是延时上报，这样可以减少阻塞。对于首屏的数据上报，可以先将数据存在变量里面，等上报组件report.js加载完毕后，再调用回调进行上报。

简单介绍RAIL模型

chrome团队提出RAIL模型概念，提供了一个思考性能问题的方法论，所以当设计师和开发者想要处理对用户影响最大的问题的时候有法可依。官方解释是：

RAIL 是一种以用户为中心的性能模型。每个网络应用均具有与其生命周期有关的四个不同方面，且这些方面以不同的方式影响着性能。

四个方面是：

  1. 加载load
    1. 在 1000 毫秒以内呈现内容，优化关键渲染路径
    2. 用户加载页面并看到关键路径内容。
  2. 空闲idle
    1. 最大程度增加空闲时间，利用空闲时间完成推迟的工作，例如，尽可能减少预加载数据，以便您的应用快速加载，并利用空闲时间加载剩余数据
    2. 主线程 JS 工作分成不大于 50 毫秒的块
    3. 用户没有与页面交互，但主线程应足够用于处理下一个用户输入。
  3. 动画animation
    1. 在 10 毫秒内生成一帧，每个帧的工作（从 JS 到绘制）完成时间小于 16 毫秒。
    2. 从纯粹的数学角度而言，每帧的预算约为 16 毫秒（1000 毫秒 / 60 帧 = 16.66 毫秒/帧）。 但因为浏览器需要花费时间将新帧绘制到屏幕上，只有 10 毫秒来执行代码。
    3. 用户滚动页面，拖动手指（例如，打开菜单）或看到动画。 拖动时，应用的响应与手指位置有关（例如，拉动刷新、滑动轮播）。 此指标仅适用于拖动的持续阶段，不适用于开始阶段。
  4. 响应response
    1. 在 100 毫秒以内响应
    2. 用户点按按钮（例如打开导航）。
